<context>

**Phase 1 Goal:** Build API infrastructure and MongoDB database with inventory management system that prevents overselling when both online shop and concert POS are selling simultaneously.

**The approach:**
This phase builds the foundation for the entire platform. The three plans are organized into two waves:

Wave 1 establishes the database layer and authentication in parallel - these are independent and can be built simultaneously. The Product schema uses per-variant version fields for optimistic locking, which is the core mechanism preventing race conditions on concurrent sales.

Wave 2 wires everything together with the Express API. The inventory routes implement atomic operations using MongoDB's findOneAndUpdate with version matching. When two transactions try to buy the last item simultaneously, only one succeeds (version matches), the other fails immediately with a 409 error (no retry, per user decision).

The audit trail is embedded directly in Order and Sale documents (not a separate collection). Every deduction logs stockBefore and stockAfter, creating a complete transaction history.

**Key insight:** The per-variant version field (rather than per-product) allows maximum concurrency. Multiple sizes/colors of the same product can be updated independently without blocking each other.

After Wave 1 completes, it will run a database connection test before proceeding to Wave 2 (this checkpoint was added during verification).
