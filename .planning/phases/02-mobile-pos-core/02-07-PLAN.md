  <name>Task 1: SyncManager and connectivity hook</name>
  <files>mobile/src/features/sync/SyncManager.ts, mobile/src/features/sync/useConnectivity.ts, mobile/src/__tests__/features/sync/syncManager.test.ts</files>
  <behavior>
    SyncManager:
    - requestSync processes pending outbox rows in batches of 10
    - Concurrent sync prevented (syncInProgress guard)
    - For sale_create type: groups sales and submits via POST /api/sales/batch
    - For sale_void type: calls POST /api/sales/:id/void
    - For sale_unvoid type: calls POST /api/sales/:id/unvoid
    - On success: marks outbox row as 'done', resets consecutiveFailures
    - On failure: increments attempt_count, sets next_attempt_at with exponential backoff (cap 30s)
    - After 3 consecutive failures: sets syncStore flag (triggers alert in UI)
    - Updates syncStore.pendingCount after each sync run
    - Updates syncStore.lastSyncAt on successful sync

    useConnectivity:
    - Subscribes to NetInfo changes
    - Updates syncStore.isOnline
    - Triggers requestSync when transitioning from offline to online

    Periodic sync:
    - startPeriodicSync starts 60-second interval
    - stopPeriodicSync clears interval
  </behavior>
  <action>
    Create mobile/src/features/sync/SyncManager.ts:
    - Follow Pattern 2 from research exactly
    - Singleton sync guard: `let syncInProgress = false`
    - requestSync(db, apiClient):
      1. If syncInProgress, return early
      2. Set syncInProgress = true
      3. Get pending outbox rows (BATCH_SIZE = 10, ordered by created_at ASC)
      4. Group sale_create entries together for batch submission
      5. Process each row:
        - sale_create: collect into batch array, submit via POST /api/sales/batch with idempotency keys
        - sale_void: POST /api/sales/:saleId/void (extract saleId from payload)
        - sale_unvoid: POST /api/sales/:saleId/unvoid
      6. On success per row: markOutboxDone, resetFailures on syncStore
      7. On failure: incrementAttempt with exponential backoff (delay = min(1000 * 2^attempt_count, 30000))
      8. If consecutiveFailures >= 3: syncStore triggers alert
      9. After processing: update syncStore.pendingCount with remaining pending count
      10. Set syncInProgress = false in finally block
    - startPeriodicSync: setInterval(requestSync, 60000)
    - stopPeriodicSync: clearInterval

    Create mobile/src/features/sync/useConnectivity.ts:
    - Follow Pattern 4 from research
    - useEffect subscribing to NetInfo.addEventListener
    - On state change: update syncStore.isOnline
    - When transitioning offline->online (wasOffline.current && now online): call requestSync
    - Show toast "X sales synced" after successful sync (per user decision)

    Write mobile/src/__tests__/features/sync/syncManager.test.ts:
    - Mock db, apiClient, syncStore
    - Test: requestSync processes pending rows and calls API
    - Test: concurrent sync prevented (second call returns early)
    - Test: successful sync marks rows as done
    - Test: failed sync increments attempt with exponential backoff
    - Test: 3 consecutive failures triggers alert flag
    - Test: sale_create entries batched together
    - Test: pendingCount updated after sync
